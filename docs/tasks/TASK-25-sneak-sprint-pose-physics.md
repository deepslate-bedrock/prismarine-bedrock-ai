## Goal

Make sneak and sprint state coherent across three surfaces:

- Local physics: sneaking/crouching changes the bot's movement, edge behavior, pose, collision dimensions, and any bridge/pathfinder-safe movement decisions that depend on crouch.
- Network output: `player_auth_input` and related movement packets carry the correct sprint/sneak inputs and pose flags at the correct ticks.
- Remote visibility: other clients/entities observe the bot entering/leaving sneaking pose and sprinting effects, including sprint particles where Bedrock expects them.

Success means focused static tests plus at least one live/recorded scenario show that a human-observed bot can sneak at a ledge without falling, expose sneaking pose to another client, and sprint with the expected visible state/effects.

## Non-Goals

- Do not finish full lateral scaffolding/pathfinder bridge support in this task unless the missing piece is directly the sneak/sprint/pose pipeline.
- Do not rewrite the Mineflayer compatibility facade except where it needs to pass through control states to native movement.
- Do not treat Geyser-only behavior as proof of Bedrock/BDS parity; use Endstone/BDS or recorded live-client traces for protocol evidence.
- Do not add unrelated entity metadata or animation refactors.

## Architecture Notes

- Bedrock sneaking must be handled as both an input/control state and an entity pose/metadata state. The task should verify whether `player_auth_input.input_data` alone is enough, or whether explicit entity metadata / actor data must also be sent or inferred by the server.
- Bridge/pathfinder behavior may require a physics-level edge constraint while sneaking, not only packet flags. A visible crouch without altered local collision/edge movement is insufficient.
- The bridge-flat e2e failure after this patch is distinct from the initial void fall. Native static physics proves sneaking clamps unsupported edges and permits supported bridge movement, and fake-world probes verify held sneak at both forward and backward edges without networking. The focused Endstone live diagnostic proves direct sneak-back auth-input is emitted, but now catches vertical fall/correction after reaching the acceptable rear AABB edge. The remaining live issue is movement/server-correction stability at a crouched ledge, then pathfinder scaffold progression.
- Runtime positions are stored as eye coordinates, but the native physics step runs in feet coordinates via `withSelfFeetPosition`. Any local pose/eye-height change before that conversion must preserve feet Y; otherwise entering sneak can move the eye-height reference and make the feet appear unsupported for subsequent ticks.
- Fake-world tests are intended for this exact class of issue: deterministic, no-network, superflat simulation with real controls and native physics, where tick count/rate can be manipulated independently of Bedrock server scheduling.
- Protocol-docs distinction: `Sneaking` is "player is sneaking" state, while `SneakDown` is the physical/input button held state. The transition bits request state changes and the server is expected to confirm/reject via `SetActorDataPacket` actor flags (`SNEAKING`, `SPRINTING`, `SWIMMING`, `GLIDING`) plus bounding-box or attribute updates. For remote visibility, do not spoof bot-sent `set_entity_data`; send complete vanilla-like input and observe the server broadcast.
- Sprint particles may be tied to sprint input flags, animation/entity events, or server-side movement state. Capture evidence before guessing.
- Other-client visibility matters: local bot state being correct is not enough if remote clients never see the sneaking pose.
